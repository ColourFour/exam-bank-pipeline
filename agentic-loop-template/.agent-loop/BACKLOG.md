# Improvement Backlog

The Planner may select from this backlog, but it may also propose a better item if the repo evidence supports it.

## Candidates

- [ ] Improve failing or weak tests.
- [ ] Remove dead code.
- [ ] Simplify confusing control flow.
- [ ] Improve one important user-facing behavior.
- [ ] Add missing validation around a risky edge case.
- [ ] Reduce generated artifact clutter.
- [ ] Improve documentation only when it directly helps usage or maintenance.

## Do not select

- Large rewrites.
- Framework swaps.
- New dependencies unless there is a strong reason.
- Cosmetic churn.
- Audit-only work unless explicitly requested.
